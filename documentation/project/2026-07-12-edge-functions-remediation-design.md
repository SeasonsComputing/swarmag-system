## 1. Problem Statement

The first Supabase Edge functions (`user-create`, `user-update`, `user-delete`,
`user-eject`) landed with a correct orchestration architecture but four classes
of defect, surfaced by strategic review on 2026-07-12:

1. **No deployment seam.** The authored functions export fetch handlers, but the
   Supabase platform requires `supabase/functions/{name}/index.ts` entrypoints
   calling `Deno.serve`. No shims, no `supabase-import-map.json`, no
   `config.toml` function registration existed.
2. **Un-settable configuration.** The edge config required custom secrets named
   `SUPABASE_PUBLIC_KEY` / `SUPABASE_SERVICE_KEY` / `SUPABASE_CLIENT_MODE`, but
   the platform reserves the `SUPABASE_` prefix and rejects such custom secrets.
3. **Unsafe synchronization semantics.** Delete/eject mutated the domain row
   before revoking the Auth identity; a failed revocation left a "deleted" user
   able to authenticate. Email-sync failure plus OTP auto-signup
   (`enable_signup = true`, no `shouldCreateUser: false`) could mint a second
   Auth identity violating the auth/domain UUID invariant.
4. **Authorization gap.** The `users` table RLS granted open `INSERT`/`UPDATE`/
   `DELETE` (`WITH CHECK (true)`), making the edge functions' administrator
   gate bypassable via direct database access.

A fifth defect was found in core during the same review: the adapter maker
dropped all falsy values from patches (D8).

## 2. Design Decisions

### D1 â€” Domain IDs come from `@core/std` `id()`

All ids are UUID v7 generated by application code via `@core/std` `id()`,
wherever creation executes â€” including inside edge functions. The database
never generates a domain id. The earlier "application-supplied UUID from
`UserCreate.id`" language is retired; `UserCreate` carries no id, and the edge
function generating the id server-side satisfies the invariant
`auth.users.id = public.users.id`.

### D2 â€” Committed entrypoint shims bridge to the Supabase platform

Each function has a committed three-line `supabase/functions/{name}/index.ts`
shim: import the wrapped handler, `Deno.serve(handler)`. Shims contain no
logic. A shim generator is deferred to the backlog until function count
justifies it.

### D3 â€” Edge configuration aliases platform-injected names

The Supabase Edge runtime injects `SUPABASE_URL`, `SUPABASE_ANON_KEY`, and
`SUPABASE_SERVICE_ROLE_KEY`. The edge config bootstrap binds the repository's
logical names (`SUPABASE_PUBLIC_KEY`, `SUPABASE_SERVICE_KEY`) to the injected
names through the `Config` alias map. `SUPABASE_CLIENT_MODE` is dropped from
the edge required list. No custom Supabase secrets are set.

### D4 â€” Access revocation precedes domain mutation

`user-delete` and `user-eject` delete the Auth identity first, then mutate the
domain row. Access revocation is the security-relevant half and must not be
the half that silently fails. `user-create` keeps its compensation (delete
Auth user when domain insert fails); a compensation failure is logged and the
original error is rethrown.

### D5 â€” Eject deletes the Auth identity; reactivation is unsupported

Setting an ejected user back to `active` does not restore an Auth identity.
Restoring access requires a future `user-reinstate` function. Ban-based
revocation (identity-preserving) is a backlog alternative.

### D6 â€” OTP login never provisions users

`sendOtp` passes `shouldCreateUser: false` and `config.toml` sets
`enable_signup = false`. User provisioning happens only through `user-create`
(service-role Auth admin, exempt from the signup switch). This closes the
mismatched-identity hazard permanently.

### D7 â€” The `users` table is read-only to direct callers

All three open write policies are dropped. RLS grants `SELECT` to
authenticated users on non-deleted rows and no write policies to any direct
caller. Domain user writes flow only through the user-management edge
functions via the service-role client after administrator authorization.

### D8 â€” Three-state adapter patch semantics _(executed 2026-07-12)_

| Patch state                                           | Meaning                | Adapter behavior |
| ----------------------------------------------------- | ---------------------- | ---------------- |
| key absent / `undefined`                              | leave column untouched | skip             |
| key present, defined value (incl. `false`, `0`, `''`) | write it               | write            |
| key present, `null`                                   | clear                  | write SQL NULL   |
| `toDomain`: column NULL                               | absent                 | omit domain key  |

`UpdateFromInstantiable` admits `null` only on optional attributes; validators
admit `null` only when `optional`; required fields therefore remain
un-clearable by construction. Fixed in `make-adapter.ts`, `protocols.ts`,
`validators.ts` (`isNullish` exported from `core/std/adt.ts`); locked by unit
tests in `source/tests/cases/`.

### D9 â€” Shared edge caller handshake in core

`core/service/edge-auth.ts` provides `makeEdgeAuth({ url, publicKey,
serviceKey })` returning `verifyCaller(request) â†’ EdgeCallerContext
{ authUserId, callerClient, serviceClient }`, throwing 401 on failure. The
platform handshake is core; the administrator predicate is domain policy and
stays in `back/supabase-edge/orchestration`.
